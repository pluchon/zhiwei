# 校园设施报修与资产管理系统第三阶段 Java 代码设计

> 本文档只明确第三阶段 Java 核心实现，不展开普通 CRUD 和伪代码。

## 1. 实现原则

- 严格遵守 `Controller -> Service -> Mapper`，请求使用 DTO，响应使用 VO。
- 业务状态、权限、资源归属、事务和并发校验由 ServiceImpl 负责。
- 资产、工单、日志、通知等一致性数据在同一事务中完成。
- SSE、OSS 删除等外部操作在事务成功后执行，失败不得回滚核心业务。
- 关键状态更新使用版本与当前状态条件更新。

## 2. 资产分类与资产台账

新增核心模块：

```text
AssetCategoryController / Service / Mapper
AssetController / Service / Mapper
AssetConverter
AssetNumberGenerator
```

核心规则：

- 资产编号按 `AST-yyyyMM-六位序号` 生成并保证唯一。
- 分类为单级分类；存在未删除资产时禁止停用。
- 资产存在未结束工单时，禁止修改、删除和停用。
- 资产删除使用逻辑删除；恢复前校验编号、分类和位置。
- 资产图片最多一张，沿用现有 OSS 权限与校验。
- 新增、修改、删除、恢复和状态变化记录操作日志或资产状态日志。

## 3. 资产关联报修

调整工单创建、编辑、提交和状态流转：

- 草稿可选择普通报修或资产报修，并可更换资产。
- 工单提交后冻结资产关联，保存资产编号、名称、分类和位置快照。
- 多个草稿可选择同一资产；提交时原子占用资产的当前工单关联，保证仅一个成功。
- 工单完成、驳回或关闭后释放资产的当前工单关联。
- 接单或派单成功后，关联资产变为维修中。
- 工单完成、驳回或关闭后，资产恢复使用中；不得覆盖管理员手动停用状态。
- 三种进入已完成的方式均形成资产维修历史；驳回和关闭不形成历史。

## 4. 管理统计与工单导出

新增管理员统计 Service：

- 按工单创建时间统计最近 7、30、90 天的故障类型、校区和楼栋分布。
- 楼栋只返回前十名，其余合并为“其他”。
- 当前状态数量实时统计，不受时间范围影响。

新增同步 `.xlsx` 导出 Service：

- 仅管理员可用，复用当前筛选条件，单次最多 1000 条。
- 固定字段导出，按创建时间倒序，不包含联系方式、评论、日志和图片。
- 文件成功生成后标记工单已导出；允许重复导出。
- 每次导出记录管理员、筛选条件、数量和结果。

## 5. 冗余附件清理

定时任务分批扫描并逐条处理：

- 未关联附件超过 24 小时后清理。
- 自动删除草稿的附件保留 7 天后清理。
- 删除前再次校验业务引用。
- 删除失败记录结果，后续任务重试；单条失败不影响整批。

## 6. 维修师傅建议与站内信

新增独立建议模块：

```text
RepairerSuggestionController / Service / Mapper
RepairerSuggestionConverter
```

- 维修师傅提交和查看本人建议；管理员查看和处理全部建议。
- 待处理建议可撤回，撤回后可编辑并重新提交。
- 撤回建议不进入管理员待处理列表，重新提交后再次进入。
- 已采纳和未采纳建议不可编辑或撤回。
- 建议提交、重新提交和处理后创建站内通知。

## 7. SSE 实时通知

新增轻量 SSE 连接管理组件：

- 使用当前服务端会话建立连接，不接收前端用户 ID。
- JWT 或会话凭证不得放入 SSE URL 查询参数。
- 同一账号支持多设备连接，每 30 秒发送心跳。
- 只推送通知变化事件，通知正文仍通过现有接口查询。
- 登录会话失效、账号停用、权限变化或退出登录时关闭连接。
- 推送失败只记录日志，不影响通知事实数据和业务事务。

## 8. 核心测试

重点覆盖：

- 资产编号唯一性、资产维护限制和恢复规则。
- 同一资产并发提交仅一个工单成功。
- 资产状态联动、快照和三种完成方式的维修历史。
- 导出权限、1000 条限制、导出标记和重复导出。
- 附件清理引用校验、失败重试和幂等。
- 建议权限、撤回编辑和管理员处理。
- SSE 用户隔离、多设备连接和会话失效关闭。
- 前两阶段核心报修流程回归。
