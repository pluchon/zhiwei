# 校园设施报修与资产管理系统需求文档 - 第三阶段

> 本文档随第三阶段需求讨论逐步完善。
>
> 仅记录已经通过讨论确认的需求；尚未确认的方案不得直接作为正式需求。

## 1. 当前阶段基础

- 第一阶段已经完成校园报修工单基础闭环、账号认证、权限控制、评论评价、附件、站内通知和审计日志。
- 第二阶段已经完善结构化报修位置、管理员手动派单、维修繁忙程度提示、长期工单提醒与自动处理、通知体验、工单查询、简单看板和核心自动化测试。
- 第三阶段继续沿用前两阶段已经确认的核心业务规则与项目强制规范。

## 2. 第三阶段总体方向

第三阶段同步推进以下两条业务主线：

- 开始建设服务于维修业务的基础资产管理能力。
- 继续深化现有报修系统能力。

两条主线可以在同一阶段并行建设，但必须明确各自范围以及资产与报修之间的关联边界，避免一次性引入过多复杂业务。

## 3. 基础资产管理

### 3.1 建设目标

第三阶段开始建设基础资产管理，但项目核心仍然是校园设施报修与维修管理。

资产管理能力主要用于帮助用户准确描述报修对象、帮助维修人员了解资产情况，并沉淀资产维修历史，不建设完整的资产全生命周期管理系统。

### 3.2 基础业务闭环

第三阶段实现以下基础资产业务闭环：

```text
管理员登记和维护资产台账
→ 用户查看资产并针对资产提交报修
→ 报修工单关联资产
→ 维修完成后形成资产维修历史
```

### 3.3 范围控制

第三阶段不实现资产二维码及扫码报修。

第三阶段暂不实现以下复杂资产管理流程：

- 采购。
- 入库审批。
- 领用。
- 借用。
- 调拨。
- 盘点。
- 折旧。
- 报废审批。
- 其他与维修业务没有直接关系的复杂资产生命周期流程。

### 3.4 资产台账基础信息

资产台账保存以下基础信息：

| 信息 | 要求 |
|---|---|
| 资产编号 | 系统自动生成，全系统唯一，格式为 `AST-yyyyMM-六位序号`，管理员不能手动修改 |
| 资产名称 | 必填 |
| 资产分类 | 必选 |
| 所属校区 | 必选，复用现有校区基础数据 |
| 所属楼栋 | 可为空，复用现有楼栋基础数据 |
| 楼层 | 可为空 |
| 房间 | 可为空 |
| 具体位置描述 | 可为空；未选择楼栋时必填 |
| 资产状态 | 必填 |
| 资产说明 | 可为空 |
| 启用日期 | 可为空 |
| 资产图片 | 可为空，最多一张 |

- 第三阶段不记录资产品牌和型号，减少老旧资产登记困难和管理员维护负担。
- 一项资产只属于一个校区和至多一个楼栋。
- 资产位置规则与现有报修位置结构保持一致。
- 资产图片沿用现有 OSS 图片格式、大小、权限和访问规则。

### 3.5 资产状态

第三阶段资产状态仅包含：

```text
IN_USE         使用中
UNDER_REPAIR   维修中
OUT_OF_SERVICE 停用
```

- “使用中（`IN_USE`）”资产允许用户提交新的关联报修。
- “维修中（`UNDER_REPAIR`）”表示资产存在正在处理的维修事项。
- “停用（`OUT_OF_SERVICE`）”资产不允许用户提交新的关联报修。

### 3.6 资产状态联动与关联工单限制

- 同一资产同一时间只能关联一个未结束的报修工单。
- 资产已经存在未结束关联工单时，不允许再次针对该资产提交新工单。
- 当前关联工单结束后，允许用户未来再次针对该资产提交报修，并形成新的维修历史。
- 关联资产的工单被维修师傅主动接单或被管理员手动派单后，资产自动变为“维修中（`UNDER_REPAIR`）”。
- 关联资产的当前工单结束后，资产自动恢复为“使用中（`IN_USE`）”。
- 关联工单被驳回或关闭后，资产自动恢复为“使用中（`IN_USE`）”，不需要管理员确认。
- 若资产已经被管理员手动停用，关联工单结束时不得自动覆盖其“停用（`OUT_OF_SERVICE`）”状态。
- 管理员可以手动将资产停用或恢复为使用中。
- 资产存在未结束关联工单时，不允许停用；管理员必须先按照工单规则关闭该工单。
- 停用资产不允许提交新的关联报修，但不影响已经提交的关联工单继续处理。
- 已停用资产的历史工单和维修历史继续正常展示。

关联工单是否结束按照以下工单状态判断：

```text
COMPLETED 已完成
REJECTED  已驳回
CLOSED    已关闭
```

- 仅当关联工单进入上述结束状态后，资产才允许关联新的报修工单。
- 资产与工单的关联及状态变更必须由后端校验，不能依赖前端限制。
- 多名用户同时针对同一资产提交报修时，只允许一个工单成功提交并关联该资产。

### 3.7 资产分类

#### 3.7.1 分类原则

- 第三阶段只建设简单、粗粒度的单级资产分类。
- 资产分类用于辅助管理员维护资产和用户查找报修对象，不追求精细化资产分类管理。
- 不建立多级分类树，不维护分类层级关系。
- 仅管理员可以新增、修改、启用和停用资产分类。
- 第三阶段不提供资产分类删除和恢复功能。
- 分类名称全系统唯一。
- 停用分类不能用于登记新资产。
- 已有资产及历史业务继续展示其分类信息。

#### 3.7.2 分类停用规则

- 管理员停用资产分类前，必须先删除该分类下的全部资产。
- 分类下仍存在未删除资产时，不允许停用该分类。
- 分类停用校验必须由后端执行，不能仅依赖前端限制。

#### 3.7.3 资产删除规则

- 资产删除统一使用逻辑删除，不物理删除资产台账和历史维修数据。
- 资产存在未结束关联工单时，不允许删除。
- 资产仅存在已经结束的历史关联工单时，允许逻辑删除。
- 资产逻辑删除后，不允许再用于新报修，也不在普通资产查询中展示。
- 资产逻辑删除后，历史工单和资产维修历史仍然保留并正常展示资产快照。
- 管理员可以恢复逻辑删除的资产。
- 恢复资产前必须重新校验资产编号唯一性、资产分类可用状态以及所属校区和楼栋关系。
- 资产恢复后默认为“使用中（`IN_USE`）”。
- 删除资产及停用分类必须记录管理员操作日志。

### 3.8 资产台账维护规则

- 仅管理员可以新增、修改、删除和恢复资产。
- 资产存在未结束关联工单时，资产全部信息均不允许修改，也不允许删除。
- 资产不存在未结束关联工单时，管理员可以修改资产基础信息。
- 历史工单始终展示提交报修时保存的资产名称、分类和位置快照，不随资产台账后续修改变化。
- 新增、修改、删除和恢复资产必须记录管理员操作日志。

### 3.9 资产查看与报修

- 管理员可以查看和维护全部资产。
- 维修师傅可以查看全部未删除资产，但不能修改资产。
- 学生和教师可以查看“使用中（`IN_USE`）”资产并针对资产提交报修。
- 资产列表支持按资产编号、名称、分类、校区和楼栋查询。
- 用户仍可不关联资产，按照现有流程提交普通位置报修。
- 关联资产报修时，系统自动带入资产位置，用户可以补充故障位置说明。

### 3.10 资产维修历史

- 资产维修历史直接来源于关联资产的报修工单，不建立需要人工填写的独立维修记录。
- 关联工单通过以下任一方式进入“已完成（`COMPLETED`）”后，形成资产维修历史：

```text
报修人主动确认完成
系统长期待确认后自动完成
管理员仲裁确认完成
```

- 未完成、被驳回或被关闭的工单不计入资产维修历史。
- 资产维修历史展示工单编号、故障类型、故障描述、维修师傅、维修结果和确认完成时间。
- 管理员和维修师傅可以查看资产的全部维修历史。
- 学生和教师只能查看自己提交的关联工单，不开放资产的全部维修历史。

### 3.11 资产维修发起边界

- 第三阶段不建立独立的资产维修任务或资产维修派发流程。
- 管理员不能直接针对资产创建维修任务。
- 管理员不能绕过报修工单直接向维修师傅派发资产维修。
- 资产维修必须由学生或教师针对资产提交报修后，进入现有报修工单流程。
- 关联资产的报修工单提交后，仍可按照现有工单规则由维修师傅主动接单或由管理员手动派单。
- 管理员主要负责资产台账、资产分类和现有报修工单的管理。

### 3.12 资产与报修工单关联方式

- 用户创建工单时可以选择“资产报修”或“普通报修”。
- 资产报修必须选择一项当前允许报修的资产。
- 普通报修不关联资产，继续使用现有报修流程。
- 草稿阶段允许切换报修类型或更换关联资产。
- 多个草稿可以暂时选择同一资产。
- 一项资产被其他工单成功提交后，其他用户选择该资产时必须看到该资产已有未结束工单的提示，并且不能继续提交关联该资产的工单。
- 多个草稿并发提交同一资产时，只允许一个工单提交成功。
- 工单提交后，不允许更换、添加或移除关联资产。
- 待匹配和待接单阶段即使允许编辑其他工单内容，也不允许修改资产关联关系。
- 被驳回工单重新转为草稿后，允许重新选择报修类型和关联资产。
- 资产报修工单保存资产编号、资产名称、资产分类和资产位置快照。
- 资产台账后续修改、停用、删除或恢复，不改变已提交工单中的资产快照。

## 4. 报修系统深化

### 4.1 范围边界

- 第三阶段不实现已接单工单的管理员转派功能。
- 维修师傅无法继续处理工单时，仍按照现有退回规则将工单重新送入匹配流程，之后管理员可以按现有规则手动派单。
- 第三阶段不实现账号人工恢复和联系方式特殊处理审批流程。
- 账号恢复与联系方式变更继续沿用现有安全规则。
- 第三阶段不统计或展示平均处理时长。
- 第三阶段不建设维修师傅排名、绩效分析或其他容易形成个人比较压力的统计功能。
- 第三阶段不建设维修师傅排班、请假和在线状态管理。

### 4.2 简单管理统计

第三阶段可以在现有管理员报修看板基础上增加以下简单统计：

- 各故障类型工单数量分布。
- 各校区工单数量分布。
- 报修数量较多的楼栋。
- 当前各工单状态数量。

- 统计内容关注设施问题和报修业务整体情况，不评价或比较维修师傅个人。
- 第三阶段不实现平均处理时长、维修师傅排名和绩效统计。
- 简单管理统计仅管理员可以查看。
- 统计支持选择最近七天、最近三十天和最近九十天，默认展示最近三十天。
- 第三阶段不提供任意时间范围统计。
- 报修数量较多的楼栋最多展示前十名，其余楼栋合并为“其他”。
- 故障类型和校区数量分布展示当前统计时间范围内的全部分类。
- 故障类型、校区和楼栋数量分布均按照工单创建时间统计。
- 当前各工单状态数量按照实时数据统计，不受最近七天、三十天或九十天范围影响。

### 4.3 冗余附件清理

- 系统通过定时任务清理 OSS 中没有有效业务引用的冗余附件。
- 上传后始终未关联工单的附件，超过二十四小时后自动清理。
- 草稿工单被自动逻辑删除后，其关联附件保留七天，再自动清理。
- 删除 OSS 文件前必须再次确认附件没有被有效业务数据引用。
- OSS 删除失败时保留失败记录，由后续定时任务自动重试。
- 每次清理必须记录成功、失败和对应的 OSS `objectKey`。
- 附件清理不影响有效工单及其历史附件。

### 4.4 简单工单导出

- 第三阶段实现面向管理需要的简单工单导出。
- 仅管理员可以导出工单。
- 导出当前筛选条件下的工单。
- 单次最多导出一千条工单；超过限制时提示管理员缩小筛选范围。
- 导出采用同步生成文件的方式，不引入消息队列、独立报表服务或异步导出任务。
- 导出文件使用 `.xlsx` 格式，并按照工单创建时间倒序排列。
- 导出文件名包含导出日期和时间。
- 导出功能必须遵守后端数据权限和敏感信息保护规则。
- 导出内容不包含评论、日志、图片和联系方式等敏感或复杂信息。
- 导出文件包含工单编号、标题、状态、故障类型、位置、资产关联信息、报修人账号与姓名、当前维修师傅账号与姓名、创建时间和确认完成时间。
- 工单首次成功导出后，记录独立的“已导出”标记。
- “已导出”不是工单业务状态，不影响工单状态流转和任何业务操作。
- 已导出的工单允许再次导出。
- 管理员工单列表支持按“已导出”和“未导出”筛选。
- 只有导出文件成功生成后，才将本次包含的工单标记为“已导出”；导出失败不得改变工单导出标记。
- 每次导出均记录操作管理员、导出时间、筛选条件、导出数量和执行结果。
- 第三阶段不建设自定义报表、复杂导出模板和定时导出功能。

### 4.5 SSE 实时通知

- 第三阶段引入轻量 SSE，为已登录用户提供站内通知实时提示。
- 第三阶段不引入 WebSocket。
- SSE 仅用于服务端向客户端单向推送通知变化事件，不承载双向通信。
- SSE 事件只提示当前用户存在新的通知或通知状态发生变化，不直接推送完整通知正文。
- 客户端收到事件后，通过现有站内通知接口刷新未读提示状态或通知列表。
- 浏览器断线后使用 SSE 原生重连能力重新建立连接。
- 用户未连接 SSE 或短暂断线时，不影响站内通知数据库记录生成和正常业务处理。
- 站内通知数据库记录仍然是通知事实来源。
- 第三阶段不引入消息队列、离线消息补偿和复杂推送确认机制。
- 用户登录成功后建立一条 SSE 连接，退出登录后关闭连接。
- 同一账号允许多个设备分别建立 SSE 连接。
- SSE 连接只允许接收当前登录用户自己的通知变化事件。
- 服务端每三十秒发送一次心跳，避免连接因长时间空闲被中间网络设备关闭。
- 登录会话失效、账号停用或权限发生变化后，服务端关闭对应 SSE 连接。
- SSE 连接建立和事件推送必须使用服务端会话校验用户身份，不能信任前端传入的用户 ID。

### 4.6 站内信与维修师傅建议

#### 4.6.1 站内信分类

第三阶段将现有站内通知入口扩展为统一站内信入口，当前包含：

```text
SYSTEM_NOTICE      系统通知
REPAIRER_SUGGESTION 维修师傅建议
```

- 系统通知继续承载工单状态变化、提醒和系统处理结果。
- 维修师傅建议用于维修师傅主动向管理员提交业务建议。
- 统一站内信入口负责分类展示和未读提示。

#### 4.6.2 建议记录与通知关系

- 维修师傅建议是需要管理员处理的独立业务记录，不能仅作为普通通知正文保存。
- 建议提交、处理和结果必须保留完整记录。
- 建议状态发生变化时，通过系统通知告知相关维修师傅和管理员。
- SSE 可以实时提示新的建议或建议处理结果，但建议正文和处理记录仍通过业务接口查询。

#### 4.6.3 权限边界

- 只有维修师傅可以提交建议。
- 管理员可以查看和处理全部维修师傅建议。
- 维修师傅只能查看自己提交的建议及其处理结果。
- 学生和教师不能提交或查看维修师傅建议。

#### 4.6.4 建议分类

维修师傅提交建议时必须选择以下分类之一：

```text
FAULT_TYPE     新增或调整故障类型
ASSET_INFO     资产信息错误或缺失
REPAIR_PROCESS 报修与维修流程改进
OTHER          其他建议
```

#### 4.6.5 建议处理

建议状态包括：

```text
PENDING  待处理
ACCEPTED 已采纳
REJECTED 未采纳
```

- 维修师傅提交建议时必须填写标题、分类和建议内容。
- 待处理建议允许维修师傅撤回。
- 建议撤回后自动进入可编辑状态，维修师傅可以修改后重新提交。
- 已采纳或未采纳的建议不能撤回或编辑。
- 管理员处理建议时必须选择处理结果并填写回复。
- 管理员采纳故障类型建议后，仍由管理员自行创建或修改故障类型，系统不自动变更基础数据。
- 建议提交和处理必须记录操作时间，并通过站内信通知相关用户。

## 5. 核心自动化测试与验收

- 资产编号生成必须保证全系统唯一。
- 同一资产并发提交报修时，只允许一个工单成功提交。
- 资产存在未结束工单时，禁止修改、删除和停用资产。
- 资产分类下存在未删除资产时，禁止停用分类。
- 资产逻辑删除与恢复规则必须通过自动化测试验证。
- 资产状态与关联工单状态联动必须正确，且不能覆盖管理员手动停用状态。
- 三种完成方式均能形成资产维修历史，驳回和关闭不得形成维修历史。
- 工单导出必须验证管理员权限、单次一千条限制、导出标记和重复导出。
- 冗余附件清理必须验证引用检查、删除失败重试和有效附件保护。
- SSE 必须验证用户身份隔离、会话失效后关闭连接和多设备连接。
- 维修师傅建议必须验证角色权限、撤回编辑和管理员处理规则。
- 第三阶段新增功能不得破坏前两阶段核心报修闭环。

## 7. 开发优先级

第三阶段按照以下顺序推进：

1. 资产分类与资产台账。
2. 资产关联报修、资产状态联动与资产维修历史。
3. 简单工单导出与管理统计。
4. 冗余附件清理。
5. 维修师傅建议与站内信分类。
6. SSE 实时通知。
7. 自动化测试与整体验收。

开发过程中可以在不产生模块冲突的前提下并行推进，但不得突破本阶段已经确认的范围边界。

## 8. 第三阶段验收基线

- 基础资产台账能够服务资产查看、关联报修和维修历史查询。
- 普通报修与资产报修均能完整使用现有工单流程。
- 资产与工单之间的并发、状态联动、快照和权限规则正确生效。
- 简单工单导出、管理统计和冗余附件清理符合本阶段规则。
- 维修师傅建议能够通过统一站内信入口提交、处理和通知。
- SSE 能够提供轻量实时通知，并保持用户身份隔离。
- 第三阶段新增功能具有与风险相匹配的自动化测试。
- 第一阶段和第二阶段已经实现的核心功能不得发生回归。
