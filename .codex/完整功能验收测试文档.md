## 1. 验收目标

本次验收从全新数据库和全新 Redis 数据开始，验证校园设施报修与资产管理系统在真实浏览器、后端接口、数据库约束和并发场景下是否形成完整、可靠的业务闭环。

核心验收目标：

1. 验证学生、教师、维修师傅、管理员四类角色的页面、接口和数据权限。
2. 验证账号认证、报修、维修、确认、仲裁、评价、通知等核心闭环。
3. 验证位置、资产、导入、统计、建议、人工恢复和 AI 辅助功能。
4. 验证状态机、事务、并发、幂等、逻辑删除、历史快照和审计日志。
5. 移除“超级管理员”概念，只保留普通管理员（`ADMIN`），管理员不得绕过角色或业务权限规则。
6. 记录全部测试证据、缺陷、修复与回归结果，最终给出可验收或不可验收结论。

### 3.1 PowerShell UTF-8 前置命令

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new()
$PSDefaultParameterValues['Out-File:Encoding'] = 'utf8'
$PSDefaultParameterValues['Set-Content:Encoding'] = 'utf8'
```

### 3.2 全新环境重置步骤

执行正式验收前停止前后端进程，清空 MySQL 与 Redis 持久化卷，再重新启动：

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new()
docker compose down -v
docker compose up -d
```

待 MySQL 与 Redis 健康后启动后端：

```powershell
Set-Location Campus-Backen
.\mvnw.cmd spring-boot:run
```

另开 UTF-8 PowerShell 启动前端：

```powershell
Set-Location front
npm run dev
```

重置后的必要检查：

- Flyway `V1` 至 `V8` 全部执行成功。
- `DevelopmentDataInitializer` 创建四个开发账号、三个故障类型、维修能力、两个校区和三个楼栋。
- 默认无工单、附件、评论、评价、通知及业务日志。
- Redis 无旧登录会话、验证码、票据或 AI 缓存。

## 6. 关键状态与验收规则

### 6.1 工单状态

| 编码 | 中文名称 | 主要允许动作 |
|---|---|---|
| `DRAFT` | 草稿 | 报修人编辑、提交 |
| `PENDING_DISPATCH` | 待匹配 | 管理员派单、驳回、关闭 |
| `PENDING_ACCEPT` | 待接单 | 维修师傅接单、管理员派单、驳回、关闭 |
| `ACCEPTED` | 已接单 | 当前维修师傅开始处理或退回 |
| `PROCESSING` | 处理中 | 当前维修师傅提交维修结果或退回 |
| `PENDING_CONFIRM` | 待确认 | 报修人确认解决或反馈未解决 |
| `PENDING_ARBITRATION` | 待仲裁 | 管理员仲裁 |
| `COMPLETED` | 已完成 | 报修人评价、符合条件时后续评价 |
| `REJECTED` | 已驳回 | 报修人重新转草稿 |
| `CLOSED` | 已关闭 | 终态 |

### 6.2 资产状态

| 编码 | 中文名称 | 核心规则 |
|---|---|---|
| `IN_USE` | 使用中 | 可用于资产报修 |
| `UNDER_REPAIR` | 维修中 | 关联工单接单或派单后自动进入 |
| `OUT_OF_SERVICE` | 停用 | 不允许新报修，工单自动联动不得覆盖管理员手动停用 |

### 6.3 验收优先级

| 优先级 | 定义 | 处理规则 |
|---|---|---|
| P0 | 核心闭环、安全、越权、数据丢失、状态错误、超级管理员残留 | 任意失败则不可验收 |
| P1 | 重要业务、日志、导出、统计、通知、AI 降级 | 修复并回归通过后方可验收 |
| P2 | 非阻断交互、文案、一般兼容性 | 可记录后限期修复 |

## 8. 重点端到端业务链路

### 8.1 普通报修完整闭环

```text
学生登录
→ 创建草稿并上传教室图片
→ 选择主校区 / 教学楼A / 水电维修
→ 提交工单
→ 维修师傅接单
→ 开始处理
→ 评论沟通
→ 提交维修结果
→ 学生确认解决
→ 学生评价
→ 双方通知、工单日志、维修记录、评价完整
```

### 8.2 未解决与仲裁闭环

```text
教师提交宿舍现场报修
→ 管理员手动派单
→ 维修师傅处理并提交结果
→ 教师连续反馈未解决
→ 第 5 次进入待仲裁
→ 管理员仲裁
→ 检查状态、历史、系统评论与通知
```

### 8.3 资产报修闭环

```text
管理员创建分类和资产
→ 学生选择资产提交报修
→ 工单接单后资产自动进入维修中
→ 工单完成后资产恢复使用中
→ 资产维修历史形成
→ 修改资产信息后历史工单仍展示原快照
```

### 8.4 管理与 AI 辅助闭环

```text
管理员导入 Excel / 图片资产
→ 审核并确认入库
→ 查看统计与导出
→ 创建相似报修数据
→ 验证疑似重复、关联推荐和派单分析
→ 验证 AI 失败降级
→ 检查 AI 审计日志
```

## 10. 验收通过标准

满足以下全部条件方可给出“验收通过”：

1. 所有 P0 用例通过，P0 缺陷数量为 0。
2. 所有 P1 缺陷修复并完成回归，或经用户明确接受剩余风险。
3. 学生/教师报修闭环、维修师傅处理闭环、管理员管理闭环、资产闭环全部通过。
4. 超级管理员概念与全权限绕过逻辑完全移除，只保留普通管理员。

8. 后端 Maven 测试通过，前端生产构建通过。
9. 外部短信、邮件、OSS、SSE 和 AI 无法真实验证的部分已有明确降级验证和剩余风险记录。
10. 最终验收报告包含用例统计、缺陷统计、修复记录、证据索引与结论。
