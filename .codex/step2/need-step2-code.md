# 校园设施报修系统第二阶段 Java 代码设计

> 本文档只说明第二阶段需求对应的 Java 核心实现。
>
> 不展开普通 CRUD、Controller 转发、字段清单和具体 SQL；未在需求中确认的功能不得自行增加。

## 0. 开发与验证环境

- 后端位于 `Campus-Backen`，使用 Java 17、Spring Boot 3、MyBatis-Plus 和 Flyway。
- MySQL 与 Redis 均使用项目根目录 `docker-compose.yml` 启动，不假设开发机安装本地 MySQL 或 Redis。
- 后端在宿主机运行时，MySQL 使用 `localhost:3307/campus_system`，Redis 使用 `localhost:6379`。
- 数据库、Redis 和后端连接信息继续通过现有 `application.yml` 默认值与环境变量配置，不在代码中写死。
- 开发或测试前先确认 Docker 服务健康；涉及数据库迁移时，由后端启动时的 Flyway 自动执行。
- 不随意清除 Docker 数据卷；需要重建或清理环境时必须先明确确认。

## 1. 核心代码原则

- 后端继续遵守 `Controller -> Service -> Mapper`。
- 工单状态流转按具体业务动作分别编写，不能使用可任意传入目标状态的通用流转方法。
- 日志、置顶系统评论、站内通知、周期开始与结束等重复操作可以统一封装。
- 权限、资源归属、账号状态和业务状态由 Service 校验。
- 接单、派单和关键状态变化由 Mapper 条件更新完成最终并发保护。
- 需求要求保持一致的状态、记录、日志、评论和通知必须处于同一数据库事务。
- 第一阶段已有流程必须保持兼容。

核心状态动作保持清晰区分，重复记录创建允许统一：

```text
业务方法：
校验权限与状态
→ Mapper 条件更新抢占工单
→ 更新成功后写入该动作要求的记录、周期、日志、评论和通知

统一辅助方法：
createOrderLog(...)
createPinnedSystemComment(...)
createNotification(...)
startOrEndCycle(...)
```

## 2. 报修位置结构化选择

### 需求对应代码

- 新增校区和楼栋相关 Entity、Mapper、Service、DTO 与 VO。
- `LocationService` 负责校区和楼栋的维护、启停、逻辑删除、恢复和报修选择列表。
- 报修选择列表返回校区和楼栋说明，供前端帮助用户正确选择。
- 工单创建与编辑请求只接收校区 ID、可空楼栋 ID、楼层、教室或房间、具体位置描述。
- `RepairOrderServiceImpl` 保存工单前，通过 `LocationService` 校验位置并取得名称和说明快照。
- 工单详情和列表展示工单快照，不根据当前校区或楼栋数据重新生成历史位置。

### 核心校验

- 校区必须启用且未删除。
- 楼栋存在时，必须启用、未删除并属于所选校区。
- 未选择楼栋时，具体位置描述必填。
- 工单接单后不能修改位置。
- 草稿、待匹配和待接单编辑位置时，重新保存最新位置快照。
- 校区下存在启用楼栋时，不能停用或删除校区。
- 存在非已完成、非已关闭关联工单时，不能删除校区或楼栋。
- 恢复前重新校验名称唯一性、所属关系和上级校区状态。

### 名称与排序

- 名称入库前去除首尾空格。
- 校区名称全系统唯一，同校区楼栋名称唯一。
- 唯一性比较忽略大小写，已逻辑删除数据继续占用名称。
- 选择列表按拼音首字母、完整名称、数据库 ID 稳定排序。

## 3. 管理员手动派单

### 核心业务方法

在管理员业务 Service 中新增明确的手动派单方法，不复用普通接单方法。

### 派单前校验

- 当前用户仍为管理员。
- 工单处于待匹配或待接单。
- 请求版本与当前工单版本一致。
- 目标用户账号正常且角色为维修师傅。
- 派单说明必填。
- 目标维修师傅不具备对应能力时，能力不匹配原因必填。

### 派单事务

同一事务内完成：

- 原子更新工单为已接单并设置当前维修师傅。
- 新增手动派单接单记录，保留管理员、派单说明和能力不匹配原因。
- 开始新的维修周期。
- 新增工单流转日志和置顶系统评论。
- 向报修人和被派单维修师傅新增站内通知。
- 新增管理员操作日志。

主动接单与管理员派单并发发生时，只允许一个操作成功。

```text
管理员派单：
重新校验管理员、工单、版本和目标维修师傅
→ 原子更新待匹配或待接单工单
→ 创建派单历史和维修周期
→ 创建日志、置顶评论、双方通知和操作日志
```

被手动派单的维修师傅继续使用第一阶段退回流程，不新增拒绝或转派功能。

## 4. 维修师傅繁忙程度

### 计算代码

- 只统计维修师傅名下的已接单和处理中工单。
- Service 将统计数量映射为空闲、适中、繁忙。
- 精确数量只用于内部计算，不写入对外 VO。

### 使用范围

- 管理员派单候选列表展示维修师傅繁忙程度。
- 维修师傅主动接单前可以查询自己的繁忙程度，用于前端展示提示。
- 学生和教师选择故障类型后，只查询整体维修力量，不返回候选维修师傅身份。
- 工单接单后，报修人可以查看实际处理维修师傅的繁忙程度。
- 没有符合条件维修师傅时仍允许提交，工单进入待匹配。

繁忙程度只用于提示，不限制主动接单或管理员派单。

## 5. 工单状态流转与周期

### 状态流转方法

以下动作分别使用独立业务方法和专属条件更新：

- 主动接单。
- 管理员手动派单。
- 开始处理。
- 提交维修结果。
- 报修人确认完成。
- 报修人反馈未解决。
- 维修师傅退回。
- 待匹配或待接单撤回为草稿。
- 已驳回重新转为草稿。
- 三十天自动完成。
- 自动完成后申请仲裁。

日志、系统评论和通知创建可以使用统一辅助方法，但每个业务动作自行决定需要生成的记录。

### 维修周期

独立记录当前维修师傅的连续维修周期。

周期规则：

- 主动接单或管理员派单成功后开始。
- 已接单进入处理中不重新开始。
- 评论和补充说明不重新开始。
- 提交维修结果进入待确认后结束。
- 报修人反馈未解决并再次进入处理中时开始新周期。
- 维修师傅退回并清空当前维修师傅后结束。

同一工单同时只能存在一个活动维修周期。

### 待确认周期

独立记录每次进入待确认的时间和提醒状态。

周期规则：

- 提交维修结果进入待确认后开始。
- 报修人确认完成、反馈未解决或自动完成后结束。
- 再次进入待确认时开始新周期并重新计时。

同一工单同时只能存在一个活动待确认周期。

### 进入草稿

- 报修人可以将自己的待匹配或待接单工单撤回为草稿。
- 报修人可以将自己的已驳回工单重新转为草稿。
- 两种动作分别校验允许的来源状态和版本，不使用普通编辑接口直接修改状态。
- 工单进入草稿后，以最新更新时间重新计算三十天自动清理期限。

### 反馈未解决

反馈未解决次数增加与状态变化必须由一次原子更新完成：

- 未达到五次时进入处理中，并开始新维修周期。
- 第五次进入待仲裁，不开始维修周期。
- 当前待确认周期结束。

```text
反馈未解决：
原子增加未解决次数并决定目标状态
→ 结束待确认周期
→ 若重新进入处理中，则开始新维修周期
→ 若进入待仲裁，则不创建维修周期
```

## 6. 三天与七天温和提醒

`RepairOrderScheduler` 负责扫描到期维修周期，具体单条处理由任务 Service 在独立事务中完成。

### 三天提醒

- 当前活动维修周期满三天时，只通知当前维修师傅。
- 同一维修周期只发送一次。

### 七天提醒

- 当前活动维修周期满七天时，通知当前维修师傅和管理员。
- 管理员列表根据当前活动维修周期已满七天展示长时间未进展标记。
- 同一维修周期只发送一次。

维修周期结束后不再满足长时间未进展条件，标记自动消失。

提醒任务通过条件更新保证多次扫描或多实例执行时不会重复发送。提醒不改变工单状态。

## 7. 待确认提醒与长期自动完成

任务 Service 按当前活动待确认周期处理：

- 满三天，只向报修人发送第一次提醒。
- 满七天，只向报修人发送第二次提醒。
- 满二十七天，只向报修人发送最终提醒。
- 满三十天且报修人仍未操作时，自动完成工单。

每项提醒和自动完成在同一待确认周期内只执行一次。

```text
定时任务：
扫描到期周期
→ 条件更新抢占单条任务
→ 抢占成功后写入通知或执行自动完成
→ 单条失败不影响后续记录
```

### 自动完成事务

同一事务内完成：

- 原子更新待确认工单为已完成。
- 记录自动完成时间。
- 结束当前待确认周期。
- 更新当前接单记录。
- 新增工单流转日志和置顶系统评论。
- 只向报修人新增站内通知。

### 自动完成后申请仲裁

- 只有报修人可以申请。
- 必须是自动完成的工单。
- 申请说明必填。
- 只能在自动完成后七天内申请。
- 成功后进入现有管理员仲裁流程。

自动完成后不能直接反馈未解决，但仍可按普通完成规则评价。

## 8. 长期未更新草稿自动清理

定时任务只处理：

- 状态为草稿。
- 未逻辑删除。
- 最后更新时间已满三十天。

单条清理使用条件更新保证只执行一次。

清理成功后逻辑删除工单并新增包含工单 ID 的审计日志，不发送通知、不删除附件记录和 OSS 文件，也不提供恢复入口。

## 9. 站内通知优化

`NotificationServiceImpl` 增加：

- 按已读状态和通知类型分页查询。
- 查询当前用户是否存在未读通知。
- 单条标记已读。
- 勾选多条批量标记已读。
- 当前用户全部通知标记已读。

核心规则：

- 所有查询和更新只能操作当前用户的通知。
- 批量请求包含他人通知时，整个请求拒绝。
- 重复标记已读保持幂等。
- 未读状态只返回是否存在未读，不返回精确数量。
- 工单相关通知继续返回工单 ID，供前端跳转。

不增加 SSE、WebSocket 或定时轮询。

## 10. 工单查询与筛选

`RepairOrderServiceImpl` 根据当前角色执行组合查询。

支持需求中明确的条件：

- 工单编号精确查询。
- 标题关键词查询。
- 创建时间范围。
- 状态、故障类型、校区和楼栋。
- 管理员按报修人账号或姓名、当前维修师傅账号或姓名查询。
- 管理员按长时间未进展筛选。
- 各角色快捷筛选。

权限规则：

- 管理员只查询正常业务数据。
- 维修师傅快捷筛选不扩大原有可见范围。
- 学生和教师只查询本人提交的工单。
- 学生和教师的已结束包含已完成、已驳回和已关闭。
- 查询结果不得泄露联系方式。
- 管理员从看板进入列表时，今日新增使用创建时间条件，今日完成使用完成时间条件。

## 11. 管理员简单报修看板

管理员业务 Service 查询需求明确的九项数据：

- 待匹配、待接单、已接单、处理中、待确认、待仲裁。
- 长时间未进展。
- 今日新增。
- 今日完成。

统计规则：

- 只有管理员可以查询。
- 排除逻辑删除工单。
- 各状态分别统计。
- 今日使用 `Asia/Shanghai` 自然日。
- 今日完成依据真实完成时间。

不增加排名、趋势、平均处理时长或导出功能。

## 12. 定时任务规则

定时任务包括：

- 维修周期三天和七天提醒。
- 待确认三天、七天和二十七天提醒。
- 待确认三十天自动完成。
- 草稿三十天自动清理。

统一要求：

- 分批扫描到期数据。
- 单条任务独立事务。
- 单条失败不影响整批。
- 使用条件更新保证幂等。
- 时间来源可在测试中控制。

## 13. 核心测试范围

必须覆盖：

- 位置选择、快照、停用、删除和恢复规则。
- 管理员派单权限、能力不匹配派单和并发派单。
- 繁忙程度与整体维修力量计算，不泄露精确数量。
- 主动接单、提交结果、确认、反馈未解决和退回流程。
- 维修周期和待确认周期的开始、延续与结束。
- 各时间节点提醒、自动完成和草稿清理不重复执行。
- 自动完成后七天仲裁期限。
- 通知归属、批量已读和全部已读。
- 各角色组合查询和快捷筛选的数据权限。
- 管理员看板统计口径。
- 第一阶段核心报修闭环回归。

## 14. 推荐实现顺序

1. 位置结构化选择和工单位置快照。
2. 工单专属状态更新与现有流程并发加固。
3. 管理员手动派单和繁忙程度。
4. 维修周期、待确认周期和相关提醒任务。
5. 自动完成、自动完成后仲裁和草稿清理。
6. 通知优化、组合查询和管理员看板。
7. 核心自动化测试与第一阶段回归测试。
