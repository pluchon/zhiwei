# 校园设施报修系统第二阶段 Vue3 前端设计

> 本文档描述第二阶段需求对应的前端调整。
>
> 继续沿用第一阶段 Vue 结构和现有页面，不自行改变后端状态、权限或业务流程。

## 0. 开发与联调环境

- 前端位于 `RuoYi-Vue3-master`，继续使用现有 Vite、Vue 3、Pinia、Element Plus 和外置同名脚本结构。
- 开发环境接口前缀继续使用 `.env.development` 中的 `/dev-api`，由 Vite 代理到宿主机后端 `http://localhost:8080`。
- MySQL 和 Redis 均由项目根目录 `docker-compose.yml` 启动，前端不直接连接数据库或 Redis。
- 联调前先确认 Docker 中的 MySQL、Redis健康，并确认后端 `8080` 已启动。
- 前端改动完成后运行现有 `npm run build:prod`，不自行更改现有环境文件、代理规则或 Vite 外置脚本转换插件。

## 1. 前端强制规范

- Vue 文件只保留 `<template>` 与外部资源声明。
- 页面脚本放在同目录同名 `.js`，存在私有样式时放在同目录同名 `.scss`。
- 保留 Vite 前置转换插件，继续支持 `<script setup src="./同名文件.js">`。
- 页面禁止直接调用 Axios，接口统一放在 `src/api`。
- 页面局部查询、表单、加载和弹窗状态不放入 Pinia。
- 页面禁止直接修改 Store 内部状态。
- 状态展示统一使用 `src/utils/repair.js` 或共享组件，页面禁止散落状态数字。
- 前端权限只控制可见性和交互，后端仍是最终权限边界。
- 关键操作提交期间禁用按钮，后端仍负责并发和幂等。

## 4. 报修创建与编辑

改造 `views/repair/create`：

- 校区使用必选单选项。
- 楼栋根据当前校区级联选择，可为空。
- 展示当前选中校区和楼栋的说明。
- 增加教室或房间输入框。
- 未选择楼栋时，具体位置描述必填。
- 不再允许自由填写校区和楼栋名称。
- 故障类型变化后查询整体维修力量。
- 没有匹配维修师傅时展示管理员跟进提示，但仍允许提交。
- 学生和教师不能选择维修师傅，也不提交维修师傅优先意向。
- 提交载荷只传位置 ID 和用户填写的位置内容，不传位置名称与说明快照。

页面切换校区时必须清空已选择楼栋，避免提交不属于当前校区的楼栋。

编辑草稿、待匹配和待接单工单时沿用同一位置表单；已接单后不展示位置编辑入口。

## 5. 校区与楼栋维护

管理员位置维护页面展示：

- 校区列表及说明、状态和删除状态。
- 选中校区下的楼栋列表。
- 新增和编辑校区、楼栋。
- 启用、停用、逻辑删除和恢复操作。

交互规则：

- 名称输入自动去除首尾空格后提交。
- 停用、删除和恢复必须二次确认。
- 删除校区前提示管理员先处理启用楼栋和关联工单。
- 删除或恢复失败时直接展示后端返回的具体原因。
- 已删除数据仅在管理员维护页面展示，不出现在报修选择列表。
- 页面不允许管理员维护排序值，列表按后端顺序展示。

## 6. 管理员手动派单

在管理员工单详情中，仅对待匹配和待接单工单展示手动派单入口。

派单弹窗展示：

- 可选择的正常维修师傅。
- 维修师傅账号和姓名。
- 繁忙程度。
- 是否具备当前故障类型能力。
- 必填派单说明。
- 能力不匹配时必填派单原因。

提交前必须二次确认，并展示：

```text
请确认已与维修师傅沟通，并确认是否继续手动派单。
```

繁忙或能力不匹配只展示提醒，不禁用提交；能力不匹配原因未填写时禁止前端提交。

派单成功后关闭弹窗并重新加载工单详情。并发冲突时提示刷新后重试。

不提供已接单工单转派、管理员替维修师傅退回或拒绝入口。

## 7. 维修师傅繁忙提示

- 管理员派单候选项展示繁忙等级，不展示精确数量。
- 维修师傅主动接单前查询本人繁忙等级。
- 当前等级为繁忙时，确认框展示“您当前接的工单较多，请慎重接取。”。
- 提示不阻止维修师傅继续接单。
- 学生和教师只查看所选故障类型的整体维修力量。
- 接单前不得展示符合条件维修师傅姓名。
- 接单后，报修人可以查看实际处理维修师傅的繁忙程度。

繁忙等级统一维护显示文本和标签样式，页面不得自行重复映射。

## 8. 工单列表与筛选

改造 `views/repair/list`，使用同一页面根据当前角色展示允许的条件。

通用筛选：

- 工单编号精确查询。
- 标题关键词。
- 创建时间范围。
- 状态、故障类型、校区和楼栋。

管理员额外筛选：

- 报修人账号或姓名。
- 当前维修师傅账号或姓名。
- 长时间未进展。

维修师傅快捷筛选：

- 我的处理中。
- 我的待确认。
- 已完成。

学生和教师快捷筛选：

- 处理中。
- 待我确认。
- 已结束。

列表规则：

- 筛选条件变化后回到第一页。
- 校区变化后清空楼栋筛选。
- 长时间未进展工单显示明确标记。
- 管理员列表展示报修人与当前维修师傅的账号和姓名。
- 学生和教师的已结束文案对应已完成、已驳回和已关闭。
- 查询条件只保存在当前页面内存，刷新或重新进入后重置。
- 不保存个人筛选偏好，不提供导出入口。

从管理员看板进入工单列表时，可以通过路由查询参数初始化本次筛选。列表读取后立即清除路由查询参数，保证用户刷新页面或重新进入时筛选条件重置。

## 9. 工单详情调整

改造 `views/repair/detail`：

- 展示完整位置名称和说明快照。
- 接单前不向学生和教师展示候选维修师傅信息。
- 接单后按第一阶段规则展示实际维修师傅信息。
- 待匹配和待接单状态向报修人展示撤回为草稿入口。
- 已驳回状态向报修人展示重新转草稿入口。
- 管理员按状态显示手动派单入口。
- 自动完成工单在七天内向报修人展示申请仲裁入口。
- 自动完成超过七天后隐藏仲裁入口。
- 自动完成工单仍展示评价入口。
- 被管理员派单的维修师傅继续显示退回入口，不显示拒绝入口。

撤回、关闭、驳回、退回、仲裁、手动派单和自动完成后申请仲裁均需二次确认或填写要求的说明。

后端返回状态或版本冲突后，页面提示并重新加载详情。

## 10. 通知页面与未读提示小点

改造 `views/notification/index`：

- 支持全部、未读、已读筛选。
- 支持通知类型筛选。
- 支持勾选多条后批量标记已读。
- 支持全部标记已读。
- 单条已读、批量已读和全部已读后立即刷新列表和未读提示小点。
- 点击工单相关通知进入对应工单详情。

通知入口只展示未读提示小点，不展示精确数量。

三天、七天、二十七天和自动完成通知继续作为普通站内通知展示；前端不自行计算提醒期限，也不显示倒计时。

未读提示小点刷新时机：

- 登录成功或刷新整个页面。
- 进入通知页面。
- 离开通知页面。
- 执行单条、批量或全部已读后。
- 普通页面切换时可刷新一次。

不使用定时轮询、SSE 或 WebSocket。用户停留在页面期间收到新通知，不要求立即出现提示小点。

未读状态可以放入轻量通知 Store，但页面只能调用 Store action 刷新，不能直接修改 Store 状态。

## 11. 管理员简单报修看板

改造管理员首页 `views/index`，只向管理员展示九项数据卡片：

- 待匹配、待接单、已接单、处理中、待确认、待仲裁。
- 长时间未进展。
- 今日新增。
- 今日完成。

交互规则：

- 进入管理员首页时查询一次。
- 提供手动刷新按钮。
- 不自动轮询。
- 点击卡片进入管理员工单列表，并初始化对应状态、日期或长时间未进展筛选。
- 今日新增卡片使用创建时间条件，今日完成卡片使用完成时间条件。
- 不展示排名、平均处理时长、趋势图和导出按钮。

其他角色首页继续沿用已有内容，不展示管理员看板。

现有首页中的“未读通知”精确数量卡片移除或改为不展示数量，避免与第二阶段未读提示小点规则冲突。

## 12. 路由与权限调整

管理员路由新增：

```text
/admin/locations
```

现有工单详情、工单列表和首页继续复用，不为不同角色重复创建页面。

按钮可见性按照角色、资源归属和状态控制：

- 只有管理员显示位置维护和手动派单。
- 只有报修人显示允许状态下的编辑、确认、反馈未解决和自动完成仲裁。
- 只有当前维修师傅显示开始处理、提交结果和退回。

按钮隐藏不能替代后端权限校验。后端拒绝后，页面以服务端结果为准。

## 13. 共享状态与组件

扩展 `src/utils/repair.js` 或现有共享组件，统一维护：

- 工单状态文本与标签类型。
- 繁忙程度文本与标签类型。
- 快捷筛选选项。
- 长时间未进展标记展示。
- 通知类型文本。

可以新增轻量共享组件：

```text
LocationSelector
BusyLevelTag
RepairFilterBar
```

只有多个页面确实复用时才抽取组件，避免为单页逻辑提前建立复杂组件。

## 14. 页面状态与交互规范

所有新增和改造页面必须处理：

```text
loading
empty
error
submitting
disabled
```

- 列表请求失败时保留筛选条件并提供重新加载。
- 表单提交失败时保留用户输入。
- 关键按钮提交期间禁用，防止重复点击。
- 二次确认文案明确说明操作后果。
- 后端返回 401、403、状态冲突和业务校验错误时展示清晰信息。
- 不在前端日志或提示中展示联系方式等敏感数据。

## 15. 前端验证

第二阶段不引入完整前端自动化测试框架。

必须完成：

- 前端生产构建通过。
- 学生和教师验证位置选择、整体维修力量、列表筛选和自动完成仲裁入口。
- 维修师傅验证繁忙提示、主动接单、退回和快捷筛选。
- 管理员验证位置维护、手动派单、长时间未进展筛选和九项看板。
- 所有角色验证通知筛选、批量已读、全部已读和提示小点刷新。
- 验证前端不展示精确工单数量、不泄露候选维修师傅身份。
- 验证无定时通知轮询、SSE 或 WebSocket 连接。

## 16. 第二阶段不实现

- 不实现资产管理、二维码、扫码报修。
- 不实现已接单工单转派。
- 不实现工单导出、排名、趋势图和复杂报表。
- 不实现精确未读数量和维修师傅精确工单数量。
- 不实现通知轮询、SSE 或 WebSocket。
- 不保存个人筛选偏好。
