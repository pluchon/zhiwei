# 校园设施报修与资产管理系统需求文档 - 第一阶段

> 一致性约定：第一阶段实现以 `need-step1-java-code.md` 为主；本文档只描述业务需求，不自行扩展后端实现或数据模型。

## 代码格式强制规范

- 禁止将多个 `import`、字段、语句、方法、HTML 标签或 CSS 规则压缩在同一行。
- 每行只表达一个清晰操作；复杂条件、链式调用和长参数必须合理换行。
- 禁止为了减少文件行数牺牲代码可读性。
- Java 重要业务逻辑必须使用中文注释说明业务原因、事务边界和并发约束。
- Java 数据库实体类及字段必须使用中文 Javadoc 说明。
- Vue 文件只保留 `<template>` 与外部资源声明，禁止内嵌 JavaScript 和 CSS。
- 每个 Vue 文件必须使用同目录同名 `.js` 文件承载脚本；存在私有样式时，使用同目录同名 `.scss` 文件承载样式。
- Vue 通过 `<script setup src="./同名文件.js"></script>` 和 `<style scoped src="./同名文件.scss"></style>` 引入脚本与私有样式。
- 由于 Vue 编译器原生禁止 `<script setup>` 使用 `src`，项目必须保留 Vite 前置转换插件，在 Vue 编译前读取并注入同名脚本；禁止移除该插件后继续使用外部脚本声明。
- 没有私有样式时不创建空 `.scss` 文件；全局共享样式继续放在 `src/assets/styles`。
- 后端调用链必须严格遵守 `Controller -> Service -> Mapper`，禁止跨层调用。
- `service.interfaces` 只声明业务接口，`service.impl` 只放同名实现类，格式为 `XxxServiceImpl implements XxxService`。
- Controller 只能声明接口注解、接收参数、调用 Service 和包装 `ApiResponse`，禁止直接依赖 Mapper、写业务逻辑、写数据库逻辑、创建业务实体、处理事务或写审计逻辑。
- Spring 管理的 Controller、Service、Component、Configuration、Filter 等类，依赖注入统一使用字段 `@Autowired`。
- 请求体必须使用 DTO，接口响应必须使用 VO 或 `PageResult<VO>`，禁止 Controller 直接使用 `Map` 接收业务请求或直接返回数据库 Entity。
- Entity 到 VO 的转换必须统一放在 `converter` 包中，使用可直接维护的手写静态转换方法，禁止依赖 MapStruct 等编译期自动生成转换实现。
- ServiceImpl 必须直接调用 converter 静态方法，禁止将 converter 注册为 Spring Bean 或通过 `@Autowired` 注入。
- ServiceImpl 禁止编写业务载荷、Redis 载荷、DTO 或 VO 内部类；此类数据结构必须抽取为独立 Java 文件并补充中文 Javadoc。
- ServiceImpl 中所有事务方法必须显式使用 `@Transactional(rollbackFor = Exception.class)`。
- VO 只描述响应字段和中文字段说明，禁止在 VO 类中手写 `from`、`toVO` 等实体转换方法。
- 重要类、重要方法和关键业务分支必须补充中文注释，说明业务原因、权限边界、事务边界和并发约束。
- 全部数据库主键必须使用 MySQL `BIGINT AUTO_INCREMENT` 自增策略，禁止雪花 ID、`assign_id` 或应用层手动生成主键。

## 1. 阶段目标

第一阶段实现校园报修工单基础闭环：

```text
学生或教师提交报修
→ 系统按故障类型匹配维修师傅
→ 维修师傅主动接单并处理
→ 报修人确认完成并评价
→ 系统记录完整流转日志
```

第一阶段后端为独立 Spring Boot 应用，前端使用 RuoYi-Vue3 外壳。

## 2. 用户角色

### 2.1 学生与教师

- 激活学校预创建账号。
- 使用账号密码或主手机号验证码登录。
- 提交、查看和跟踪自己的报修工单。
- 确认维修完成或反馈未解决。
- 提交评价和一次追评。

### 2.2 维修师傅

- 查看当前能力规则匹配的待接工单。
- 主动接单并查看自己已接工单。
- 开始处理、提交维修结果、退回无法处理的工单。
- 与报修人在工单内评论沟通。

### 2.3 管理员

- 查看和筛选全部工单。
- 驳回无效报修、关闭异常工单、处理仲裁。
- 查看工单流转日志。
- 创建和维护学生、教师、维修师傅账号。
- 调整普通用户角色、状态和联系方式。
- 维护故障类型、维修能力关系和原因字典。

管理员不能创建管理员、将普通用户提升为管理员或修改其他管理员。

## 3. 数据通用规范

- 数据库使用 MySQL 8 和 `utf8mb4`。
- 业务数据默认不物理删除。
- 业务表包含 `create_time`、`update_time`、`delete_state`。
- `delete_state` 使用 `0` 表示正常、`1` 表示已删除。
- 普通业务查询排除已逻辑删除数据。
- 工单关闭使用业务状态 `CLOSED`，不使用逻辑删除表达。
- 工单日志和安全日志原则上只允许新增。
- 关键业务操作必须记录操作人、操作时间和操作日志。

## 4. 登录与权限

### 4.1 账号范围

- 学校预先创建学生和教师账号，登录账号为学号或工号。
- 学生和教师首次使用初始密码时必须激活账号。
- 维修师傅和管理员不开放自行注册。
- 未激活账号不能使用报修业务，也不能自助找回密码。
- 停用或封禁账号不能登录、激活或自助恢复。
- 第一阶段每个用户仅拥有一个角色。

固定角色：

```text
STUDENT
TEACHER
REPAIRER
ADMIN
```

### 4.2 联系方式

- 主手机号必须已验证且全系统唯一，可用于登录、恢复和换绑。
- 验证邮箱可选且全系统唯一，可用于账号恢复和主手机号不可用时的身份验证。
- 家长联系电话仅作备注，不用于登录、恢复或身份验证。
- 联系方式入库前必须标准化。

主手机号换绑：

```text
当前手机号可用：验证当前手机号 + 验证新手机号
当前手机号不可用：验证已绑定邮箱 + 验证新手机号
手机号和邮箱均不可用：普通用户由管理员特殊处理
```

验证邮箱：

```text
首次绑定：验证当前主手机号 + 验证新邮箱
更换邮箱：验证当前邮箱 + 验证新邮箱
当前邮箱不可用：验证主手机号 + 验证新邮箱
```

- 验证邮箱不允许直接解绑，只允许绑定或换绑。
- 联系方式变更成功后，系统清除该账号全部登录会话并要求重新登录。
- 管理员特殊处理必须记录操作日志。

### 4.3 行为验证码与验证码发送

第一阶段使用 `tianai-captcha 1.4.1`。发送手机或邮箱验证码前，用户必须先通过行为验证码。

```text
获取 Tianai 行为验证码挑战
→ 完成人机验证
→ 获取一次性 captchaTicket
→ 根据业务场景消费 captchaTicket：
   - 密码登录
   - 请求发送短信或邮箱验证码
```

规则：

- `captchaTicket` 有效期 2 分钟，只能使用一次。
- Tianai 行为验证码结果仅用于换取业务 `captchaTicket`，不能直接发送短信或邮件验证码。
- `captchaTicket` 绑定业务场景和目标地址，可用于密码登录行为验证或授权发送验证码。
- 验证码有效期 5 分钟，只能成功使用一次。
- 单个验证码最多校验 5 次，达到上限后失效。
- 验证码绑定使用场景和目标地址，不能跨场景使用。
- 同一场景和目标地址 60 秒内只能发送一次。
- 同一场景和目标地址 15 分钟内最多发送 5 次。
- 第三方发送成功或失败均计入限流次数。
- 找回账号等敏感入口使用模糊响应，避免暴露联系方式是否绑定账号。

### 4.4 账号激活

```text
提交 userNo 和初始密码
→ 校验账号未激活且状态正常
→ 验证学校预留手机号
→ 设置与初始密码不同的新密码
→ 激活账号
→ 返回登录页
```

- 停用或封禁账号不能开始或完成激活。
- 激活成功后不创建登录会话、不签发 JWT。
- 重复完成激活时返回“账号已激活”。

### 4.5 登录

支持：

- `userNo + 密码 + LOGIN_PASSWORD captchaTicket` 登录。
- 主手机号验证码登录。

密码登录限制：

- 账号和请求 IP 分别统计失败次数。
- 同一账号连续密码错误 5 次后，禁止密码登录 15 分钟。
- 登录成功后清除账号密码错误次数。
- 行为验证码错误不计入密码错误次数。
- 密码登录锁定不影响验证码登录、激活和恢复。

### 4.6 账号恢复与修改密码

- 已激活且状态正常的账号可通过主手机号或验证邮箱恢复。
- 主手机号和邮箱均不可用时，普通用户联系管理员处理。
- 邮箱恢复后设置“主手机号待确认”提醒，不限制登录和业务使用。
- 新密码不能与当前密码相同。
- 修改或恢复密码成功后，清除全部登录会话并要求重新登录。
- 恢复成功后不签发 JWT。

### 4.7 JWT 与服务端会话

- 登录成功后签发 JWT，JWT 仅保存随机 `sessionId` 和标准时间声明。
- Redis 保存服务端会话，有效期固定为 14 天，不自动续期。
- 允许同一账号同时登录多个设备。
- 每次访问受保护接口时，系统查询数据库并实时校验激活状态、账号状态、角色和 `securityStamp`。
- 密码、联系方式、角色或账号状态变化后，系统更新 `securityStamp` 并清除全部会话。
- 任何权限判断由后端完成，不能信任前端传入的用户 ID 或角色。

### 4.8 认证接口

```text
POST /auth/captcha/challenge
POST /auth/captcha/ticket
POST /auth/verification-codes
POST /auth/activation/start
POST /auth/activation/complete
POST /auth/login/password
POST /auth/login/phone
POST /auth/recovery/verify
POST /auth/recovery/complete
PUT  /auth/password
PUT  /auth/contacts/phone
PUT  /auth/contacts/email
POST /auth/logout
GET  /users/me
```

通用响应：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "traceId": "7d45c8..."
}
```

### 4.9 安全日志

记录：

- 登录成功与失败。
- 密码修改与恢复。
- 联系方式换绑和管理员特殊处理。
- 用户停用、封禁和角色调整。

日志中的手机号和邮箱必须脱敏，禁止记录密码、验证码、JWT 和业务票据明文。

## 5. 报修工单

### 5.1 报修人和快照

- 仅学生和教师可以提交报修，不允许匿名提交。
- 工单保存报修时的角色、昵称和头像快照。
- 用户后续修改个人信息时，历史工单快照不自动变化。
- 对其他普通用户仅展示报修人昵称和头像。

### 5.2 故障类型与维修能力

- 故障类型由管理员新增、修改、启用和停用。
- 停用类型不能用于新工单，历史工单正常展示。
- 管理员维护维修师傅与故障类型的能力关系。
- 第一阶段根据账号状态和能力关系动态匹配维修师傅。
- 第一阶段不保存候选维修师傅结果。

### 5.3 报修位置与表单

工单直接保存：

| 字段 | 要求 |
|---|---|
| 校区 | 必填 |
| 楼栋 | 可为空 |
| 楼层 | 可为空 |
| 位置补充说明 | 必填，手动填写具体方位 |

报修表单：

| 字段 | 要求 |
|---|---|
| 标题 | 必填 |
| 故障类型 | 必填，必须启用 |
| 故障描述 | 必填 |
| 报修位置 | 必填 |
| 联系电话 | 必填，仅用于当前工单 |
| 现场图片 | 提交时 1～5 张 |

- 草稿阶段图片可以为空。
- 工单联系电话不修改账号绑定手机号。
- 提交接口必须具备幂等性。

### 5.4 工单状态

```text
DRAFT               草稿
PENDING_DISPATCH    待匹配
PENDING_ACCEPT      待接单
ACCEPTED            已接单
PROCESSING          处理中
PENDING_CONFIRM     待确认
PENDING_ARBITRATION 待仲裁
COMPLETED           已完成
REJECTED            已驳回
CLOSED              已关闭
```

主流程：

```text
PENDING_ACCEPT
→ ACCEPTED
→ PROCESSING
→ PENDING_CONFIRM
→ COMPLETED
```

提交与异常流转：

```text
DRAFT → PENDING_ACCEPT：存在匹配维修师傅
DRAFT → PENDING_DISPATCH：不存在匹配维修师傅
PENDING_DISPATCH → PENDING_ACCEPT：后续存在匹配维修师傅
PENDING_DISPATCH / PENDING_ACCEPT → DRAFT
PENDING_DISPATCH / PENDING_ACCEPT → REJECTED
REJECTED → DRAFT
ACCEPTED → PENDING_DISPATCH
PENDING_CONFIRM → PROCESSING
第 5 次反馈未解决 → PENDING_ARBITRATION
异常工单 → CLOSED
```

### 5.5 报修人操作

| 状态 | 允许操作 |
|---|---|
| `DRAFT` | 编辑、提交 |
| `PENDING_DISPATCH` | 编辑、撤回 |
| `PENDING_ACCEPT` | 编辑、撤回 |
| `ACCEPTED` | 新增补充说明 |
| `PROCESSING` | 查看进度 |
| `PENDING_CONFIRM` | 确认完成、反馈未解决 |
| `PENDING_ARBITRATION` | 查看仲裁进度 |
| `COMPLETED` | 查看、评价 |
| `REJECTED` | 查看原因、转为草稿 |
| `CLOSED` | 查看关闭原因 |

- 待匹配或待接单阶段修改工单时增加版本并记录日志。
- 接单成功后，报修人不能继续修改正文或撤回。

### 5.6 动态匹配与接单

提交工单时：

```text
存在符合条件的维修师傅 → PENDING_ACCEPT
不存在符合条件的维修师傅 → PENDING_DISPATCH
```

待接列表条件：

```text
工单状态为 PENDING_ACCEPT
+ 维修师傅账号正常
+ 维修师傅具有对应故障类型能力
```

- 接单时重新校验账号状态、能力关系、工单状态和版本。
- 多人同时接单时，仅第一名成功。
- 新增或删除维修能力关系后，系统重新检查相关待匹配和待接单工单。
- 维修师傅账号恢复正常或变为不可用时，系统重新检查受影响工单。
- 第一阶段不提供管理员手动派单。

### 5.7 维修处理

- 当前维修师傅接单后，工单进入 `ACCEPTED`。
- 当前维修师傅开始处理后，工单进入 `PROCESSING`。
- 开始处理时向报修人发送短信，短信失败不影响状态变更。
- 当前维修师傅提交维修结果后，工单进入 `PENDING_CONFIRM`。
- 维修结果说明可为空，结果图片可为空且最多 5 张。
- 报修人确认解决后，工单进入 `COMPLETED`。
- 第 1～4 次反馈未解决后，工单返回 `PROCESSING`。
- 第 5 次反馈未解决后，工单进入 `PENDING_ARBITRATION`。

### 5.8 退回

- 当前维修师傅无法处理时，可以选择预设原因并退回。
- 选择其他原因时必须填写说明。
- 退回后清空当前维修师傅，工单重新进入匹配流程。
- 原维修师傅的历史记录和评论保留，但不能继续新增评论。
- 退回原因写入工单日志并生成置顶系统评论。

### 5.9 管理员操作

驳回：

- 仅可在工单尚未接单时执行。
- 必须选择原因，其他原因需要填写说明。
- 驳回后进入 `REJECTED`。

关闭：

- 用于恶意报修、重复且无需处理、报修人申请终止、设施已拆除或其他无法继续流转的工单。
- 必须填写关闭原因。
- 关闭后进入 `CLOSED`。

仲裁：

- 管理员查看日志、维修结果、评论和反馈记录。
- 可以恢复原维修师傅继续处理、重新进入匹配流程、确认完成或关闭。
- 必须填写处理说明。

管理员操作必须生成工单日志、置顶系统评论和站内通知。

### 5.10 并发与日志

- 工单使用 `version` 字段实现乐观锁。
- 编辑、接单和关键状态变更使用状态与版本条件更新。
- 工单状态更新、接单记录、工单日志、系统评论和站内通知在同一数据库事务中完成。
- 状态时间和原因从工单日志查询。

## 6. 评论、评价、附件与通知

### 6.1 评论

- 报修人和当前维修师傅可以发表普通评论。
- 评论发送后 12 小时内允许本人撤回。
- 撤回仅隐藏正文，保留发送人与时间并记录操作日志。
- 管理员可以删除或置顶评论，删除使用逻辑删除并记录日志。
- 系统评论不可由普通用户撤回或删除。

### 6.2 评价

- 仅报修人可以评价自己的已完成工单。
- 评分为 1～5 星。
- 评价内容可为空。
- 首次评价后不允许修改。
- 允许追加一次追评，追评提交后不允许修改。
- 评价对象为最终处理工单的维修师傅。

### 6.3 图片附件

- 报修现场图片和维修结果图片存储到 OSS。
- 第一阶段图片通过后端校验并上传。
- 单张图片不超过 20 MB。
- 仅允许后端可识别的常见图片格式。
- 数据库存储 OSS `objectKey`，访问时生成临时签名 URL。
- 报修现场图片关联草稿工单，维修结果图片关联对应维修记录。
- 后端校验当前用户对工单和图片操作的权限。

### 6.4 站内通知

- 接单、开始处理、提交结果、反馈未解决、退回、驳回、关闭、仲裁和确认完成时生成站内通知。
- 用户可以分页查看通知，并将通知标记为已读。
- 站内通知与工单数据库操作保持事务一致。
- 短信和邮件失败不回滚工单事务，只记录失败日志。

## 7. 基础数据与查询

管理员维护：

- 故障类型。
- 维修师傅与故障类型能力关系。
- 学生、教师和维修师傅账号。
- 退回、驳回和关闭原因字典。

字典规则：

- `OTHER` 表示其他原因，选择时说明必填。
- 业务记录保存最终原因文本，不保存字典 ID。
- 停用字典不能用于新业务，不影响历史展示。

列表权限：

- 报修人只能查看自己提交的工单。
- 维修师傅可以查看动态匹配的待接工单和自己已接工单。
- 管理员可以查看全部工单。
- 工单列表支持分页，并按状态、故障类型、校区和楼栋筛选。

## 后端分层、Spring 注入与 DTO/VO 强制规范

- 后端调用链必须严格遵守 `Controller -> Service -> Mapper`，禁止跨层调用。
- `service` 包只允许放业务接口；同名实现类必须放在 `service.impl` 包中，并使用 `XxxServiceImpl implements XxxService`。
- Controller 禁止直接依赖 Mapper，禁止写业务逻辑、数据库逻辑、事务逻辑、审计逻辑和实体组装逻辑。
- Controller 仅允许声明接口注解、接收 DTO 参数、调用 Service、返回 `ApiResponse`。
- Spring Bean 依赖统一使用字段 `@Autowired` 注入。
- 所有请求体必须使用 DTO，禁止使用 `Map<String, Object>` 或 `Map<String, String>` 直接接收业务请求。
- 所有接口响应必须使用 VO 或 `PageResult<VO>`，禁止直接返回数据库 Entity。
- DTO 只表达请求参数，VO 只表达前端需要的响应字段；敏感字段如密码摘要、`securityStamp` 禁止出现在 VO。
- 重要类、重要方法和关键业务分支必须补充中文注释，说明业务原因、权限边界、事务边界和并发约束。
