# Campus-System Codex 长期执行守则

> 本文件是项目长期约束清单，属于不可随意突破的开发底线。
>
> 每次开始需求分析、设计、编码、修复或评审前，必须先阅读本文件，再阅读当前阶段的需求文档。

## 1. 需求与范围纪律

- 当前阶段需求文档是实现范围基线，禁止自行增加未确认的业务流程、表、字段、状态、角色或接口。
- 实现优先级遵守当前阶段文档中的 `P0 -> P1 -> P2`；不得为了增强项影响核心功能交付。
- 修改已有功能时必须保持历史数据和第一阶段流程兼容，除非需求明确要求迁移或废弃。
- 当前阶段明确不做的功能不得偷偷混入实现。
- 需求文档之间发生冲突时：
  1. 优先遵守当前阶段明确指定的主设计文档。
  2. 其次遵守更具体、更严格的约束。
  3. 无法判断且会影响数据模型、权限或业务流程时，先向用户确认。
- 不得只完成表面页面或接口；需求中声明的权限、事务、日志、并发、测试和验收条件都属于功能的一部分。

## 2. 代码可读性强制规范

- 禁止将多个 `import`、字段、语句、方法、HTML 标签或 CSS 规则压缩在同一行。
- 每行只表达一个清晰操作。
- 复杂条件、链式调用和长参数必须合理换行。
- 禁止为了减少文件行数牺牲代码可读性。
- Java 类型统一在文件顶部使用 `import`，业务代码中禁止散落 `java.util.xxx` 等全限定类名。
- Java 数据库实体类及字段必须使用中文 Javadoc。
- Java 重要类、重要方法和关键业务分支必须使用中文注释说明业务原因、权限边界、事务边界和并发约束。
- 注释用于解释原因和约束，不写重复代码含义的无效注释。
- Java 注释风格统一遵守以下规则：
  - **类注释**：使用单行 `// 中文说明`，紧贴类声明上方；不使用 Javadoc `/** */`。
  - **字段 / 属性注释**：使用单行 `// 中文说明`，紧贴字段声明上方；不使用 Javadoc `/** */`。
  - **Controller 方法注释**：使用块注释 `/** 中文说明 */`，一句话说明接口用途。
  - **ServiceImpl、Mapper 等非 Controller 方法**：需要注释时使用单行 `//` 或块注释均可，视复杂度而定。
- 每个 DTO、VO 和 Entity 必须是独立的顶层 Java 文件，禁止在一个文件中用静态内部类聚合多个 DTO 或 VO。

## 3. Java 后端分层底线

- 后端调用链必须严格遵守：

```text
Controller -> Service -> Mapper
```

- 禁止 Controller 直接依赖 Mapper。
- Controller 只能负责接口注解、接收参数、调用 Service 和包装 `ApiResponse`。
- Controller 禁止编写业务逻辑、数据库逻辑、事务逻辑、审计逻辑、权限逻辑或实体组装逻辑。
- `service.interfaces` 只声明业务接口。
- `service.impl` 只放同名实现类，格式为 `XxxServiceImpl implements XxxService`。
- ServiceImpl 负责业务流程、资源归属校验、权限校验、状态校验、事务和必要的 Redis 操作。
- Spring 管理的 Controller、Service、Component、Configuration、Filter 等类，依赖注入统一使用字段 `@Autowired`。
- 基础 CRUD、分页、逻辑删除和时间填充优先使用 MyBatis-Plus。
- 状态条件更新、接单并发、资产状态联动等需要原子判断的场景使用明确的条件更新或手写 SQL。
- 所有业务状态变化必须通过明确的 Service 方法完成，禁止使用普通 CRUD 接口直接修改状态。

## 4. DTO、VO、Entity 与转换规范

- 所有请求体必须使用独立 DTO。
- 禁止使用 `Map<String, Object>`、`Map<String, String>` 等 Map 直接接收业务请求。
- 所有接口响应必须使用 VO 或 `PageResult<VO>`。
- 禁止 Controller 直接返回数据库 Entity。
- DTO 只表达请求参数。
- VO 只表达前端需要的响应字段和中文字段说明。
- 密码摘要、`securityStamp` 等敏感字段禁止出现在 VO。
- VO 内禁止编写 `from`、`toVO` 等实体转换方法。
- Entity 到 VO 的转换统一放在 `converter` 包，使用可直接维护的手写静态方法。
- 禁止引入 MapStruct 等编译期自动生成转换实现。
- ServiceImpl 必须直接调用 converter 静态方法，禁止将 converter 注册为 Spring Bean 或注入。
- ServiceImpl 禁止编写业务载荷、Redis 载荷、DTO 或 VO 内部类；必须抽取为独立 Java 文件并补充中文 Javadoc。

## 5. 数据库与数据安全规范

- 全部数据库主键统一使用 MySQL `BIGINT AUTO_INCREMENT`。
- 禁止雪花 ID、`assign_id` 和应用层手动生成数据库主键。
- 新增业务表默认包含：

```text
create_time
update_time
delete_state
```

- `delete_state`：`0` 表示正常，`1` 表示已删除。
- 业务数据默认使用逻辑删除，普通查询必须排除已逻辑删除数据。
- 业务关闭状态必须使用业务状态表达，禁止用逻辑删除代替。
- 已被历史业务引用的数据不得物理删除。
- 工单日志、安全日志、资产状态日志等关键历史记录原则上只允许新增。
- 关键字段必须建立必要的唯一约束和查询索引，不能只依赖应用层校验。
- 唯一字段入库前必须标准化；手机号、验证邮箱等全系统唯一规则必须同时由业务校验和数据库约束保护。
- Flyway 迁移只能向前演进，不得随意修改已投入使用的历史迁移文件。
- 生产环境不得初始化默认管理员密码或开发演示账号。

## 6. 权限与安全边界

- 后端是最终安全边界，前端权限仅用于改善交互体验。
- 禁止信任前端传入的用户 ID、角色、资源归属或业务状态。
- Service 必须从当前认证上下文获取用户身份，并校验角色、资源归属、账号状态和具体操作权限。
- 数据库是账号状态、角色和权限的事实来源。
- 未认证返回 401，无权限返回 403。
- 普通用户只能查看和操作需求明确允许的资源。
- 管理员不能创建管理员、将普通用户提升为管理员或修改其他管理员。
- 停用、封禁或未激活账号必须遵守需求中的登录和业务限制。
- 上传、下载、查看或删除附件前，必须校验当前用户对关联业务资源的权限。
- 二维码、导出文件、接口响应和日志不得泄露不必要的敏感信息。

## 7. 认证、票据与会话规范

- Tianai 行为验证码只用于换取业务 `captchaTicket`，两者职责不可混用。
- 验证码和业务票据必须绑定业务场景和目标地址，禁止跨场景使用。
- 一次性票据必须原子消费，防止并发重复使用。
- 激活、恢复、换绑等流程采用“先占用票据，事务成功后删除”的规则。
- 密码、联系方式、角色或账号状态变化后，必须更新 `securityStamp` 并清除该账号全部会话。
- 清除 Redis 会话和删除票据应在数据库事务成功后执行。
- 日志禁止记录明文密码、验证码、JWT、业务票据、完整手机号和完整邮箱。
- 登录和联系方式等安全操作必须记录安全日志或操作日志。

## 8. 事务、并发与幂等规范

- ServiceImpl 中所有事务方法必须显式使用：

```java
@Transactional(rollbackFor = Exception.class)
```

- 数据库事务内只处理必须保持一致的数据库数据。
- Redis 清理、短信、邮件、SSE 和其他外部系统调用原则上在数据库事务成功后执行。
- 短信、邮件、SSE 等外部通知失败不得回滚核心业务事务，只记录失败日志或执行后续补偿。
- 状态更新、业务记录、流转日志、系统评论和站内通知等需求声明需一致的数据，必须处于同一数据库事务。
- 工单和资产等核心状态变更必须校验当前状态与版本。
- 编辑、接单和关键状态变化必须防止并发覆盖。
- 接单时必须重新校验维修师傅账号状态、能力关系、工单状态和版本。
- 提交、自动任务和其他可能重复触发的入口必须具备幂等性。
- 定时任务必须防止多实例重复执行；单条失败不能中断整批任务。
- 删除数据或 OSS 文件前必须再次确认没有任何有效业务引用。

## 9. 工单与资产业务底线

- 只有学生和教师可以提交报修，不允许匿名提交。
- 工单权限、允许操作和状态流转必须严格遵守状态机，禁止非法跨状态修改。
- 接单后报修人不能继续修改工单正文、关联资产或撤回工单。
- 工单关闭使用 `CLOSED`，不使用逻辑删除。
- 退回、驳回、关闭、仲裁等关键操作必须填写规定原因或说明，并生成要求的日志、系统评论和通知。
- 选择 `OTHER` 等其他原因时，说明必须填写。
- 历史工单、维修记录、评论、快照和日志不得因用户、位置、分类或资产信息变化而失真。
- 停用的分类、字典、位置或资产不能用于新业务，但历史数据必须正常展示。
- 资产 `INACTIVE` 和 `SCRAPPED` 状态不得被工单自动联动覆盖。
- 同一资产存在其他未结束工单时，不得错误恢复为 `NORMAL`。
- 资产状态自动变化和管理员手动变化都必须记录资产状态日志。

## 10. 日志与审计规范

- 关键业务操作必须记录操作人、操作时间、目标资源和操作结果。
- 登录成功与失败、密码和联系方式变化、用户状态与角色变化必须记录安全日志。
- 工单状态变化必须记录工单流转日志。
- 管理员特殊处理、导入、导出、删除和状态修正必须记录操作日志。
- 日志中的手机号和邮箱必须脱敏。
- 禁止在日志中记录密码、验证码、JWT、业务票据和其他认证秘密。
- 日志内容应能支持问题追踪，但不得泄露敏感数据。

## 11. Vue 前端强制规范

- Vue 文件只保留 `<template>` 与外部资源声明，禁止内嵌 JavaScript 和 CSS。
- 每个 Vue 文件必须使用同目录同名 `.js` 文件承载脚本。
- 存在私有样式时，使用同目录同名 `.scss` 文件；没有私有样式时不创建空 `.scss`。
- Vue 文件通过以下方式引用资源：

```html
<script setup src="./同名文件.js"></script>
<style scoped lang="scss" src="./同名文件.scss"></style>
```

- 必须保留 Vite 前置转换插件，在 Vue 编译前读取并注入同名脚本。
- 禁止移除转换插件后继续使用 `<script setup src>`。
- 页面禁止直接调用 Axios，统一通过 `src/api` 封装。
- 页面禁止直接修改 Store 内部状态。
- 工单和资产状态统一使用枚举映射或共享组件，禁止在页面散落状态数字和重复映射。
- 前端路由、按钮和页面可见性按角色控制，但后端仍必须再次鉴权。
- 关闭、驳回、退回、仲裁、删除等关键操作必须二次确认。
- 提交期间禁用重复操作按钮，但后端仍必须负责幂等和并发控制。
- 异步页面必须正确处理加载、空数据、错误和无权限状态。

## 12. 测试与验收纪律

- 每次修改必须评估对第一阶段认证、报修、接单、仲裁、评价和通知流程的回归风险。
- 新增核心业务不能只依赖应用启动测试，必须补充与风险匹配的自动化测试。
- 权限、状态机、事务、并发、幂等、逻辑删除、唯一约束和历史兼容属于重点测试范围。
- 修复缺陷时应尽量增加能够复现问题的测试。
- 后端修改后至少运行相关 Maven 测试。
- 前端修改后至少运行生产构建检查。
- 涉及数据库迁移、MySQL、Redis、OSS 或并发行为时，应执行相应集成验证；无法执行时必须明确说明。
- 完成任务前必须对照当前阶段验收标准复核，不能只确认代码能够编译。

## 13. Codex 工作要求

- 开始任务前先阅读本文件和当前阶段需求文档。
- 与用户讨论需求、展示流程和说明业务状态时，优先使用中文状态名称，方便用户区分；技术设计必须引用状态编码时，使用“中文名称（状态编码）”形式。
- 修改前先检查现有实现和项目约定，优先延续已有结构。
- 不得绕过规范以追求更快实现。
- 不得删除、回退或覆盖用户已有改动。
- 发现现有代码违反本文件且与当前任务直接相关时，应在本次修改中一并修正；无关问题记录但不扩大修改范围。
- 发现需求、数据模型、权限或状态规则存在高风险歧义时，先停止相关实现并向用户确认。
- 完成后必须说明修改内容、验证结果和尚未验证的风险。
